# SuperSXO — Technical Risk Register

## Purpose

This register documents known technical risks to the repository, build pipeline, and publication surface of SuperSXO.com. Each risk entry includes severity classification, prevention mechanisms, detection methods, and response procedures.

Machine-readable version: `data/technical-risk-register.json`  
Validator: `scripts/validate_security_baseline.py`

---

## Risk Format

Each entry follows this structure:

- **Risk ID:** Unique identifier (RISK-NNN)
- **Category:** Risk domain
- **Description:** What the risk is and how it could manifest
- **Severity:** high / medium / low
- **Prevention:** How the risk is prevented architecturally or procedurally
- **Detection:** How the risk is detected if prevention fails
- **Response:** How to respond if the risk materializes
- **Blocks Publication:** Whether this risk must be resolved before any route is published

## Risk Register

### RISK-001 — Accidental Public Page Generation

**Category:** build_boundary  
**Severity:** high  
**Description:** A public HTML page is generated and added to the repository or deployed before any route has been approved for publication.  
**Prevention:** `scripts/build.py` checks route status before generating output. No route currently has `status: published`.  
**Detection:** `scripts/validate_build_boundaries.py` — fails if root `index.html` or `output/` directory exists. `scripts/validate_repository_hygiene.py` — confirms `output/` is absent.  
**Response:** Delete generated file(s). Review build script invocation. Investigate how the build was triggered without route authorization. Log incident in `DECISION_LOG.md`.  
**Blocks Publication:** Yes

---

### RISK-002 — Root index.html Created Before Approval

**Category:** build_boundary  
**Severity:** high  
**Description:** A root `index.html` file is created at the repository root, making it deployable as a public home page before route publication approval.  
**Prevention:** Architecture enforces that HTML output is only generated by `scripts/build.py` when a route has `status: published`. No public HTML files are authored manually outside the template system.  
**Detection:** `scripts/validate_build_boundaries.py`; `scripts/validate_repository_hygiene.py`.  
**Response:** Delete `index.html`. Review git history to identify origin. Log incident in `DECISION_LOG.md`.  
**Blocks Publication:** Yes

---

### RISK-003 — output/ Directory Created Before Build Authorization

**Category:** build_boundary  
**Severity:** high  
**Description:** The `output/` directory is created before the build pipeline is authorized, potentially containing unpublished or untested public pages.  
**Prevention:** Build script requires `status: published` on routes before writing to `output/`. No routes are currently published.  
**Detection:** `scripts/validate_build_boundaries.py`; `scripts/validate_repository_hygiene.py`.  
**Response:** Delete `output/` directory. Review build script invocation log. Log incident in `DECISION_LOG.md`.  
**Blocks Publication:** Yes

---

### RISK-004 — Route Status Changed to Published Prematurely

**Category:** route_governance  
**Severity:** high  
**Description:** A route in `data/routes.json` has its `status` changed to `published` before all publication gates have been passed and operator approval given.  
**Prevention:** Route governance rules in `ROUTE_GOVERNANCE.md`. All routes currently locked at `status: planned`.  
**Detection:** `scripts/validate_routes.py` — validates route status and structure.  
**Response:** Revert route status to `planned`. Log incident in `DECISION_LOG.md`. Identify which publication gates remain unmet before re-attempting publication.  
**Blocks Publication:** Yes

---

### RISK-005 — JavaScript Introduced Before Approval

**Category:** javascript_policy  
**Severity:** high  
**Description:** A `.js` file is added to the repository, or inline JavaScript or event handlers are inserted into HTML templates, before JavaScript has been explicitly approved.  
**Prevention:** No `.js` files exist in the repository. No `<script>` tags with JavaScript in templates. JavaScript restriction policy documented in `SECURITY_BASELINE.md`.  
**Detection:** `scripts/validate_repository_hygiene.py` — scans for `.js` files and inline event handlers (`onclick=`, `onload=`, `onerror=`, `onmouseover=`).  
**Response:** Remove JavaScript files. Remove inline handlers from templates. Log incident in `DECISION_LOG.md`.  
**Blocks Publication:** Yes

---

### RISK-006 — External Scripts Introduced Before Approval

**Category:** third_party_policy  
**Severity:** high  
**Description:** An external script tag (`<script src="http...">`), CDN-hosted library, or tracking pixel is added to an HTML template without operator approval.  
**Prevention:** No `<script src="http">` tags in templates. Third-party script restriction policy in `SECURITY_BASELINE.md`.  
**Detection:** `scripts/validate_repository_hygiene.py` — scans HTML template files for external script tags.  
**Response:** Remove external script tags immediately. Assess what data the script may have accessed or transmitted. If a tracking or analytics script was exposed, log full incident in `DECISION_LOG.md`. Consider whether credential rotation is needed.  
**Blocks Publication:** Yes

---

### RISK-007 — Heavy 3D or Canvas-Only Content Introduced

**Category:** interface_policy  
**Severity:** medium  
**Description:** WebGL rendering, Three.js, canvas-only content, or other heavy 3D is introduced, making the asset depend on JavaScript for content delivery and weakening accessibility and search indexability.  
**Prevention:** Interface governance in `INTERFACE_GOVERNANCE.md` prohibits canvas-only rendering. Spatial interface is progressive enhancement only per `TECHNICAL_STANDARD.md`.  
**Detection:** `scripts/validate_repository_hygiene.py` — scans HTML templates for `<canvas` elements and 3D library references.  
**Response:** Replace canvas or 3D elements with semantic HTML equivalents. Log design decision in `DECISION_LOG.md`.  
**Blocks Publication:** No

---

### RISK-008 — Accessibility Weakened by Spatial Interface

**Category:** accessibility  
**Severity:** medium  
**Description:** Spatial design tokens, visual depth effects, or component patterns weaken keyboard accessibility, screen reader compatibility, or WCAG AA compliance.  
**Prevention:** Component registry enforces accessibility requirements for every registered component. Semantic HTML structure is required.  
**Detection:** `scripts/validate_visual_system.py` — checks component accessibility requirements are defined. Manual audit required before publication.  
**Response:** Restore semantic HTML structure. Review component accessibility requirements in `data/component-registry.json`. Log remediation in `DECISION_LOG.md`.  
**Blocks Publication:** No

### RISK-009 — SEO Weakened by Non-Semantic Prototype

**Category:** seo_integrity  
**Severity:** medium  
**Description:** A non-public prototype containing non-semantic markup, placeholder headings, or incomplete structure is accidentally included in the build output, harming SEO on publication.  
**Prevention:** Prototypes are locked to `non_public_prototype` status. Build script will not generate output from prototype files. Prototype validator enforces non-public status.  
**Detection:** `scripts/validate_prototypes.py` — verifies prototype source files are non-public; `scripts/validate_build_boundaries.py` — confirms no output exists.  
**Response:** Confirm prototype remains non-public. If output was accidentally generated from prototype, delete it. Log in `DECISION_LOG.md`.  
**Blocks Publication:** No

### RISK-010 — GitHub Actions Workflow Permissions Too Broad

**Category:** workflow_policy  
**Severity:** high  
**Description:** The GitHub Actions workflow has write permissions or excessively broad access, enabling a compromised workflow or supply-chain attack to modify the repository, publish releases, or exfiltrate secrets.  
**Prevention:** Workflow permissions restricted to `contents: read`. No third-party actions beyond official `actions/checkout` and `actions/setup-python`.  
**Detection:** Manual review of `.github/workflows/*.yml` files. No automated check currently deployed.  
**Response:** Restrict permissions to `contents: read`. Audit the workflow run history for unauthorized actions. Log incident in `DECISION_LOG.md`.  
**Blocks Publication:** Yes

---

### RISK-011 — Secrets Committed to Repository

**Category:** repository_secrets  
**Severity:** high  
**Description:** An API key, access token, private key, `.env` file, or other live credential is committed to the repository, exposing it to anyone with repository access or git history access.  
**Prevention:** No `.env` files in repository. No hardcoded keys in any file. All secrets stored in deployment platform environment variables only.  
**Detection:** `scripts/validate_repository_hygiene.py` — scans for `.env` files by filename and secret marker patterns in YAML files.  
**Response:** Rotate the exposed credential immediately before any other action. Remove from git history using `git filter-branch` or BFG Repo Cleaner. Assess whether public exposure occurred. Log full incident in `DECISION_LOG.md`.  
**Blocks Publication:** Yes

---

### RISK-012 — Unapproved Dependencies Added

**Category:** dependency_policy  
**Severity:** high  
**Description:** A `requirements.txt`, `package.json`, `pyproject.toml`, or other dependency manifest is added, introducing external packages without review of vulnerability surface, maintenance status, or bundle size.  
**Prevention:** All scripts use Python standard library only. No package managers approved. Dependency restriction policy in `SECURITY_BASELINE.md`.  
**Detection:** Manual review for dependency manifest files. `scripts/validate_repository_hygiene.py` scans for unapproved JavaScript files that would indicate an npm-based dependency.  
**Response:** Remove dependency manifest. Remove installed packages. Document why the dependency was needed and explicitly approve it in `DECISION_LOG.md` before reintroduction.  
**Blocks Publication:** Yes

---

### RISK-013 — Duplicate Route or Broken Internal Link

**Category:** route_governance  
**Severity:** medium  
**Description:** A duplicate route path is added to `data/routes.json`, or an internal link in a template or navigation file references a path not registered in the route registry, creating orphaned pages or broken navigation.  
**Prevention:** Route registry validates path uniqueness. Navigation validation confirms all nav links are registered routes.  
**Detection:** `scripts/validate_routes.py` — checks for unique route paths; `scripts/validate_navigation.py` — checks nav links against route registry.  
**Response:** Remove duplicate route entry. Fix broken internal link. Log in `DECISION_LOG.md` if a published route was affected.  
**Blocks Publication:** No

---

### RISK-014 — Monetization Script Introduced Before Approval

**Category:** monetization_policy  
**Severity:** high  
**Description:** An affiliate link, payment processing script, advertising code, or conversion tracking pixel is introduced to an HTML template or route before monetization has been approved through the governance gate.  
**Prevention:** Monetization rules in `data/monetization-rules.json`. Only routes with `monetization_allowed: true` in `data/routes.json` may carry commercial elements, and only after passing the monetization quality gate.  
**Detection:** `scripts/validate_repository_hygiene.py` — scans HTML templates for affiliate, stripe, paypal, and adsbygoogle references.  
**Response:** Remove monetization script or link immediately. Log incident in `DECISION_LOG.md`. Confirm which routes have monetization approval before reintroducing any commercial elements.  
**Blocks Publication:** Yes
